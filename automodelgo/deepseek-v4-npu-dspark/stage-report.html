关键观察

1. PIECEWISE 是负优化: DSpark+PIECEWISE 32并发 1179ms 比无 DSpark 的 688ms 还慢 1.7x。根因是 PIECEWISE 在 MoE 层 graph break，单请求 batch=1 走 eager 回退 (300ms+)，DSpark draft+verify overhead 超过 accept 收益。

2. FULL 是正确路径: FULL_DECODE_ONLY 单请求 48-57ms (无 DSpark)，加 DSpark 后 43-58ms。num_spec=3 是最优点 (accept 2.25 最高)。

三、特性开发原理

3.1 DSpark 算法原理

DSpark 是 DeepSeek 提出的推测解码方法，核心区别于传统 EAGLE：draft model 是半自回归的——并行 backbone 一次产出 N 个 logits，Markov head 逐位置注入偏置后采样。

传统推测解码:
  Target prefix → Draft 逐 token 生成 → Target verify → accept/reject
  瓶颈: draft 阶段仍逐 token 串行

DSpark:
  Target prefix → Parallel backbone (一次出 N 个 logits)
                → Markov sequential bias + sample
                → Target 一次验证全部
  优势: draft 阶段并行 (仅轻量 Markov 采样串行)

三个核心设计

A. 并行 Backbone (架构)

Draft model 嵌入在 target checkpoint 的 mtp.{0,1,2}.* 中，复用 target decoder layer 架构。额外显存 ~20-30% (3 层 decoder + draft KV cache)。

DSparkDraftModel:
├── embed_tokens      → alias target (零额外显存)
├── lm_head           → alias target (零额外显存)
├── main_proj         → target hidden states → draft 维度 (12288→4096)
├── main_norm         → RMSNorm
├── layers × 3        → 复用 target DeepseekV2DecoderLayer (MLA + MoE + hc_pre/post)
├── norm              → head 前 RMSNorm
├── hc_head           → FP32 小参数
└── markov_head       → DSpark 专属 (~16MB)
    ├── markov_w1: [V × 256]  嵌入查表
    └── markov_w2: [256 × V]  logit 偏置投影

B. 非因果 Attention (attention 层)

DSpark query[i] 能看到 block 内所有 query token (包括 i 之后的)，让 block 内信息充分流动。Ascend DSA kernel 硬编码 cmp_mask_mode=3 (causal)，PR #12003 解法：在外部预计算每个 query 的可见 KV 物理 slot 列表，作为显式索引 tensor 传入。

C. Markov Head (采样层)

并行 backbone 的弱点：各位置 logits 独立计算，缺少 token 间依赖。Markov head 用 rank=256 的 vocab×vocab bigram 转移矩阵分解，注入上一个 token 的偏置。

base_logits[i] = lm_head(hidden[i])               ← 并行 backbone
bias[i] = markov_w2(markov_w1(prev_token[i-1]))    ← 低秩 bigram 转移
logits[i] = base_logits[i] + bias[i]
token[i] = gumbel_sample(logits[i])

七、经验总结

7.1 三条核心原则

原则 1: 先理解，再动手

最大浪费 (~28h) 源于"跑起来再说"——启动→崩→修→重启→崩→修。每次 restart cycle 至少 7 分钟 (deploy 3min + 加载 3min + 触发 1min)。架构理解推迟到第 12 个 bug 之后，导致前 11 个 bug 有 4 个是单一根因的衍生品。正确节奏: Day 0 花 4 小时读 PR + 画数据流图 + 列约束矩阵。

原则 2: 一手证据，不二手推断

v11/v12 在二手描述上反复推断 Bug #10，v13 读一手日志发现实际错误完全不同。v18 在交接文档推断上假设 num_spec=4 可用，v19 读源码证伪。phase summary 是摘要，不是真相。每次卡住时，回到 run 子目录的一手堆栈。

原则 3: 基线先行，隔离问题域

v14→v15 的策略转向直接将 DSpark 调试范围从"全部组件"压缩到"draft 路径"。target 的正确性一旦证明，后续每个错误都有单一归属。图模式阶段重复了同一模式: 先跑 FULL baseline (无 DSpark, 57ms)，再叠加 DSpark。

7.2 可复用检查清单

前置可行性校验 (动手前必做)

  • 模型大小 vs 单节点 HBM → 是否需要多节点？ (567GB > 512GB → DP=4)
  • vllm.__version__ 实际值 vs 代码假设值 → 版本分支是否正确？
  • CLI 参数 vs ModelConfig 默认值 → 位置参数是否被覆盖？ (--model 被 --served-model-name 覆盖)
  • num_spec+1 是否整除 TP？ (可行集 {0,1,3,7} for TP=8)
  • enable_sp() 实际返回值 → 是否被 flashcomm1/shared_expert_dp 强制？

调试纪律 (每个 bug)

  • 先看完整 traceback + 堆栈，不只看最后一行就动手
  • 同时检查所有可能维度 (TP/DP/SP)，不逐个排除
  • 问"这个错误合理吗？" → 不合理先查前置条件
  • 全局扫描同类问题 (如所有 reduce_scatter 点)，不修一个暴露一个
  • 死锁用 py-spy dump，不用静态推断

7.4 关于"循环感"

用户在 v13 阶段反馈"一直在解决问题，陷入循环了"。事后看不是循环，而是 bug 层层暴露的正常过程。但"循环感"来源真实：

  1. phase summary 二手描述累积误差 — 每份 summary 推断的根因都偏了一点，后续推断越偏越远。v13/v19 两次用一手日志/源码打断链条。
  2. 资源冲突背景噪声 — master 节点 NPU 被他人占用，让人分不清是代码 bug 还是环境问题。
  3. restart cycle 等待 — 每次部署 7 分钟起跳，反馈速度慢加剧"打地鼠"感。

解决"循环感"的核心操作: 一手日志 + 基线先行。两次应用都实现了从"似乎循环"到"完全跑通"的突破。
