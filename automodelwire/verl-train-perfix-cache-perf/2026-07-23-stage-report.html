① 摘要与关键指标

在 Qwen3-8B (Dense) + Ascend 910B + VeRL GRPO 上验证 PrefixSharing (prefix-0501) 的训练加速效果。 跨 ppo_micro_batch_size_per_gpu ∈ {1,2,4,8} × {baseline, PrefixSharing} 共 9 组实测（3 步/组）， 并定位+修复了前缀匹配率低下的根因。

核心发现

1. PS 集成端到端跑通 — patch 激活 → forward_step 100% 覆盖 → 运行时调用链完整，机制正确
2. PS 收益受两层限制 — ① dispatch 是否保持同 prompt 同组（已修复，匹配率 3.7×）；② prompt 占序列比例（数据集决定，GSM8K 短 prompt 上限仅 6.6%）
3. 在 GSM8K 上 PS 无净加速 — dispatch 修复后匹配率达理论值 72%，但 prompt 仅 84 token，PS 节省的计算被自身开销抵消
4. HBM 显存线性可预测 — 系数 0.85-0.88 GB/mb，mb=8 实测 35.65GB（64GB 卡安全），推翻早期"mb=8 OOM (86GB)"的误判

⑤ Q3 — PS 匹配率根因与 dispatch 修复

5.1 问题: 匹配率仅 1.28-2.39%，远低于预期

GRPO 的 N-rollout (n=8) 天然提供共享前缀（同 prompt 的 8 条 rollout 共享 prompt 本身）: gen_batch.repeat(repeat_times=8, interleave=True) → 同 prompt rollout 物理连续 [p0×8, p1×8, ...]； rollout 前按 prompt 注入 uid，repeat 后同 prompt rollout 共享 uid。 理论匹配率 = prompt 占比 ≈ 84/(84+1197) ≈ 6.6%（GSM8K），但实测仅 1.28-2.39%。

5.2 根因: DP 负载均衡重排打散同 prompt rollout

verl 的 DP 负载均衡重排（ray_trainer.py:_balance_batch，balance_batch=True 时执行）打散了同 prompt 的连续块:

  • use_prefix_grouper=False（默认）→ get_seqlen_balanced_partitions（Karmarkar-Karp 长度均衡）+ partition.sort(key=workload)（按 seq_len 排序）
  • 同 prompt 连续块 [p0×8, p1×8, ...] 被按长度重组 → DP 分片 + chunk 后每个 micro-batch 是多条"长度相近的不同 prompt"
  • 只共享 chat 模板前缀（3-21 token），而非完整 prompt（84 token）
证据: 旧组 expected_reused_prefix_tokens_per_layer 多数 3-10（chat 模板），少数 62-194（碰巧同 prompt）。完整 prompt ≈84 token，说明大部分微批次没保留同 prompt 分组。

5.3 修复: 仅改 dispatch，不碰 attention

直接改 verl overlay 源码（ray_trainer.py），引入 KEEP_PROMPT_GROUP_DISPATCH 环境变量:

  • dispatch 走 get_group_balanced_partitions（按 uid 保持同 prompt 同组）
  • 跳过长度排序（否则 partition.sort 又打散 uid 分组）
  • 与 use_prefix_grouper 解耦: 不触发 attention patch（monkey_patch.py:324）、不 import 外部 prefix_grouper 包——保持 PS 加速对比的干净单变量
为何不开 use_prefix_grouper=true: 该标志捆绑三件事（dispatch 分组 + attention patch + 外部依赖），引入额外性能/代码路径影响点，污染对比。直接改源码只取 dispatch 子集更可控。

5.5 第二层上限: GSM8K prompt 太短

L14 早期用 max_prompt_length=1024（配置 padding 上限）估算 ~33% 匹配率，错把上限当实际:

理论值达成率—72%

PS 收益的两层限制模型

层 1 — dispatch 分组（可控）: 不修复则同 prompt rollout 被打散，匹配率 ~1.4%；修复后达理论值 72%
层 2 — prompt 占比（数据集决定）: 短 prompt 数据集上限低（GSM8K 6.6%），长 prompt 数据集才有可能显著加速
两层都满足才有效益。 GSM8K 仅满足层 1（修复后），层 2 不满足 → PS 在 GSM8K 上能省的 forward 计算仅 ~6.6%，被自身开销（前缀检测/裁剪/restore）抵消，净加速 ≈ 0。

ID教训严重度

L11registry.py 同模块多 patch 覆盖 bug: forward_step 和 vocab_parallel_log_probs 共用 transformer_impl → dict 按模块名去重覆盖（上游 bug）→ lookup 改为 list[PatchSpec]forward_step 未 patch

L14GRPO N-rollout 天然提供共享前缀: repeat_interleave 产生同 prompt 连续块，"数据集是伪命题"（算法性质保证，无需构造数据）。但收益层面需 prompt 够长方向纠偏

核心诊断教训 (L13-L16): 集成验证要区分"框架跑通"与"功能生效"两层。PS 唯一硬约束是 micro_batch≥2；诊断"加速不及预期"要先量化核心机制指标（匹配率），而非只看端到端 step 时间；配置开关常捆绑多个副作用（use_prefix_grouper 同时改 dispatch + attention），实验对比时要拆解出干净的单变量；估算理论上限要用数据集实际统计量不能用量化上限。

⑧ 结论与下一步

主要结论

  1. PrefixSharing 在 VeRL GRPO 上端到端集成成功 — 机制正确（forward_step 100% 覆盖、运行时调用链完整、训练无 NaN）
  2. dispatch 修复有效 — KEEP_PROMPT_GROUP_DISPATCH 让同 prompt rollout 进同一 micro-batch，匹配率 1.28%→4.73%（3.7×），复用完整 prompt 主体
  3. 在 GSM8K 上 PS 无净加速 — prompt 仅 84 token（占序列 6.6%），PS 节省的计算被自身开销抵消。这是数据集特性决定的硬上限，非集成 bug
  4. HBM 显存线性可预测 — 系数 0.85-0.88 GB/mb，mb=8 实测 35.65GB，64GB 卡安全。推翻早期 OOM 误判
  5. micro_batch↑ 是基线最优调优方向 — mb=1→8 step ↓34%（640→421s），HBM 性价比最高的区间是 mb=4（442s, 32GB）
  6. PS 显著加速需长 prompt 数据集 + dispatch 修复配置 — prompt 占比 30%+ 时 PS 可能有可观净加速

下一步建议

  • 换长 prompt 数据集验证 PS 显著加速 — 带长 system prompt 的 RL 任务（>1K prompt），预期 prompt 占比 30%+ → PS 理论收益可观
  • mb=4 是最优性价比基线 — step 442s、HBM 32GB (余量 32GB)、吞吐 182 tok/s/GPU。mb=8 step 421s 但 HBM 已接近安全线（35.65/64GB），后续长上下文时可能 OOM
  • G3-G5 前缀合并 + prefix-0501 组合 需先实现上下文压缩模块；G4 是短上下文下的最优理论组合
  • FSDP 路线 (A1/A2) 优先级低于 Megatron 路线 — G1 已验证 Megatron 可用且性能稳定，FSDP 需额外解决 PrefixGrouper 与 remove_padding/dynamic_bsz 的不兼容问题
