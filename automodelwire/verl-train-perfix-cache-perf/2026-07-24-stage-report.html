③ Q1 — PS 在 10K 下的加速效果

研究问题: 10K 上下文下 PS 是否净加速? 加速来自哪个阶段?

3.1 actor_update 加速 (PS 主要作用点)

PS patch 作用于 Megatron forward_step, 影响 actor_update / old_log_prob / ref 三阶段训练前向。actor_update (forward+backward) 是最重阶段, 也是 PS 收益最大处。

mbbaseline (s)PS (s)加速比HBM 节省
2472.7269.21.76× (+76%)+0.6 GB
4445.0152.42.92× (+192%)+8.1 GB
8OOM95.2PS 独跑通 (baseline OOM)—
加速比随 mb 单调增大 1.76× → 2.92× → (mb8 仅 PS 可跑)。根因不是计算密度提升 (baseline actor_update 几乎不随 mb 变, 472→445s 仅 -6%, 因 10K 下 gen 占主导), 而是 dispatch 匹配率提升: mb 增大让同 prompt 的 8 个 rollout 更易聚集进同一 micro-batch, reuse_ratio 47%→73%→84%。

infer (old_log_prob + ref, forward_only) 阶段 PS 反慢 20-24%: torch_ref 后端的 KV 拼接 + prefix 规划开销 > KV 复用节省。这是 torch_ref (纯 PyTorch 参考实现) 的固有开销, NPU 优化的 flash_atten_npu 后端理论上可消除 (见 ⑦)。

3.4 完整 step 加速 (含 gen/rollout, 真实可部署)

step_profile 实测 PS gen ~185s 且跨 mb 近恒定 (mb2=184.6 / mb4=182.7 / mb8=187.0s) — 证实 PS 不影响 gen (vLLM rollout 与训练前向解耦)。完整 step = gen + 2×infer + actor_update + update_weights。

mbbaseline full (s)PS full (s)全 step 净加速vs 前向加速
21000.5865.61.16×1.20×

8OOM699.6PS 独跑通—
全 step 加速 (1.16-1.28×) 略低于前向加速 (1.20-1.37×), 因 gen (~185s, 训练无关) 在 baseline/PS 共用且占 step 19-27%, 稀释了训练前向收益。这是真实可部署加速——含 rollout 的完整 step。注: step_profile 阶段不含 ref, 完整 step 由 timing_records 的 infer (=old_log_prob+ref) + actor_update + step_profile 的 gen/update_weights 组合计算。

⑤ Q3 — mb=8 可行性与 HBM 斜率不对称

5.1 baseline_mb8 OOM

NPU out of memory. Tried to allocate 12.87 GiB
(NPU 3; 60.96 GiB total; 35.58 GiB already allocated; 12.93 GiB free)

actor_update 阶段需额外 ~13GB (activation + KV), 已用 35.58GB + 12.87GB > 60.96GB 上限。

5.2 HBM 外推 vs 实测 (推翻 "PS HBM ≈ baseline" 误判)

修正记录: 初版报告据 mb2/mb4 差值 <1GB 判 "PS HBM ≈ baseline → ps_mb8 外推同 OOM, 未实测"。此判据错误——应看 PS 自身增长斜率, 而非与 baseline 的差值。实测推翻外推。
mbbaseline actor peak (GB)PS actor peak (GB)baseline 斜率PS 斜率
245.9345.34——
456.3948.33+10.46+2.99
8OOM (实测)50.29 (实测跑通)外推 66.85 OOM ✓外推 51.32, 实测 50.29 ✓
  • baseline HBM 斜率 +10.46 GB/2倍mb (activation 随 mb 线性涨, reuser 无裁剪) → mb8 外推 66.85 > 64 → OOM (实测确认)
  • PS HBM 斜率 +2.99 GB/2倍mb (reuser 的 KV 计算被裁剪, 仅 prefix-maker 保留完整 KV) → mb8 外推 51.32, 实测 50.29GB, 余量 13.7GB 跑通 ✓
根因: PS 在 10K 下并非 "HBM ≈ baseline", 而是 HBM 增长远慢于 baseline (斜率比 3.5:1)。mb2 时两者相近 (45.93 vs 45.34) 造成 "≈" 错觉, 但随 mb 增大 baseline 的 activation 主导增长, PS 的 reuser 裁剪抵消这部分, 差距拉开 (mb4 已差 8GB, mb8 差 >16GB)。教训: HBM 可行性外推必须基于各自实测斜率, 不能用某一点差值推断 "两者相等"。

5.3 结论

mb=8 可行性不对称: baseline_mb8 OOM (HBM 硬约束), ps_mb8 跑通 (50.29GB, 余量 13.7GB, actor_update 95.2s)。PS 不仅加速训练, 还解锁了 baseline 不可行的高并发档位——这是 PS 在 10K 下的双重价值 (省时 + 省显存 + 解锁并发)。

⑥ 三层限制模型 (GSM8K / 10K / 50K)

结合三组实验, PS 收益受三层限制; 10K 是首组三层同时满足的甜区。

限制层GSM8K (短)10K50K (长)状态
L1 Dispatch1.28%→4.73% (修复)47-84%49.4%✅ 已修复
L2 Prompt 占比6.6% (无收益)49-84% (甜区)49.4% (足够)✅ 10K/50K 满足
L3 内存容量29-36GB ✅baseline 45-56 / PS 45-5067GB 🔴OOM⚠️ 10K mb≤4 双可行, mb8 仅 PS
10K 是 PS 的甜区: L1 已修复 + L2 满足 + L3 PS 在 mb≤8 可行 → 首次实现净加速 (1.20-1.37× 前向) 且解锁 mb8 高并发档。50K 因 L3 内存硬约束 OOM, 不可行。

⑦ flash_atten_npu 后端发现 (额外)

10K 下测试了 flash_atten_npu 后端 (50K 时此步 FlashBackendValidationError 崩溃)。

维度50K10K
layer=1 attention🔴 API 错误崩溃✅ 成功执行
后续 layer/backward—🔴 hang (死锁)
新发现: flash_atten_npu 在 10K (shape 小) 下 layer=1 成功 (50K 大 shape 触发 npu_fusion_attention API 不兼容), 但后续 layer 在同步点死锁 (main_ppo state=S, CPU_delta=0, 日志停 7.7min)。10K shape 治愈了 API 错误但暴露了 kernel 内部 hang。torch_ref 因此仍是 10K 下唯一可用后端; 若修复 hang, infer 阶段 20-24% 损失可望消除, 净加速有望达 1.5-2×。

⑧ 结论与下一步

  1. PS 在 10K 上下文下首次实现净训练加速: 前向 1.20×(mb2)/1.37×(mb4), 含 gen 完整 step 1.16×/1.28×。此前 GSM8K 无收益、50K OOM。
  2. 加速比随并发度单调增大: mb2 1.76× → mb4 2.92× (actor_update), reuse_ratio 47%→73%→84%。
  3. PS 降低 HBM 峰值且增长远慢于 baseline: 斜率 +3 vs +10.5 GB/2倍mb (3.5:1), mb4 省 8.1GB。
  4. mb=8 可行性不对称: baseline_mb8 OOM, ps_mb8 跑通 (50.29GB, 余量 13.7GB, actor_update 95.2s)。PS 解锁 baseline 不可行的高并发档。
  5. 10K 是 PS 甜区: L1 dispatch 已修复 + L2 prompt 占比满足 + L3 内存 PS 在 mb≤8 可行, 三层同时满足。
  6. flash_atten_npu 10K layer=1 成功但后续 hang, torch_ref 仍是唯一可用后端。
  7. 外推判据修正: HBM 可行性必须基于各自实测斜率, 不能用单点差值判 "两者相等" (初版误判 mb8 双 OOM 的根因)。

下一步建议

  • 修复 flash_atten_npu hang → 测 PS NPU 后端 10K 净加速 (预期 > torch_ref, 消除 infer 20-24% 损失)
  • 测 20K 上下文 mb{2,4} → 确认甜区上界 (L3 内存边界)
  • ps_mb8 长训练 (>3 步) 确认稳态 reuse 84% 与 HBM 50GB 稳定
